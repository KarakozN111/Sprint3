# Sprint 3 — Batch ETL + Airflow

##Общее описание проекта
В рамках данного проекта реализован отказоустойчивый пакетный (Batch) пайплайн обработки данных для образовательной платформы. Архитектура решения построена на оркестрации Python-скриптов с помощью Apache Airflow. Пайплайн автоматизирует процесс извлечения данных из плоских файлов (CSV), их поэтапную загрузку, валидацию и трансформацию до аналитических витрин.

**Ключевая особенность:** проект реализован без использования систем потоковой передачи данных (без Debezium и Kafka). Вся логика опирается на периодические пакетные запуски.

---

## Архитектура и слои данных

Движение данных организовано по классической схеме многослойного хранилища (МХД):

1. **Источники (Source Layer):** статические и динамические выгрузки в формате `.csv`, расположенные в локальной директории контейнера Airflow (`/opt/airflow/datasets/`).
2. **Сырой слой (Raw Layer):** схема `raw` в PostgreSQL. Сюда данные попадают напрямую из файлов без изменения структуры.
3. **Слой хранилища (DWH Layer):** Схема `dwh`. Место, где данные очищаются, дедуплицируются и приводятся к единому формату.
4. **Слой витрин (Mart Layer):** Схема `mart`. Финальные агрегаты и таблицы, готовые для построения отчетов и BI-инструментов.

## Стратегии загрузки данных

Для оптимизации ресурсов БД и времени работы пайплайна применены две разные стратегии загрузки:

### 1. Full Refresh (Полное обновление)
* **Применяется для:** Таблицы `students` (справочник студентов).
* **Логика работы:** Поскольку справочник студентов относительно небольшой, но может содержать скрытые изменения в старых записях, применяется метод полной перезаписи. Перед каждой загрузкой вызывается команда `TRUNCATE ... RESTART IDENTITY CASCADE`. Это гарантирует, что таблица полностью очищается, сбрасываются счетчики ID, а затем данные записываются "с нуля" методом `append`.

### 2. Incremental Load (Инкрементальная / дельта-загрузка)
* **Применяется для:** Таблицы `student_marks` (динамический лог оценок).
* **Логика работы:** Таблица оценок может расти бесконечно, поэтому перегружать её целиком неэффективно.
  * Скрипт делает запрос в базу данных и находит максимальную дату изменения: `SELECT MAX(updated_at) FROM raw.student_marks;`.
  * Читается свежий CSV-файл, и из него отфильтровываются только те строки, у которых `updated_at` строго больше, чем дата из БД.
  * Если база была пустой, загружается весь файл. Если новые строки найдены, они дописываются в конец таблицы (метод `append`). Если обновлений нет, скрипт пропускает шаг записи.

## Мониторинг прогонов и отказоустойчивость

### Статусная таблица `raw.etl_dag_runs`
Для контроля за состоянием пайплайна и быстрого поиска ошибок внедрен внутренний аудит. Каждая итерация работы DAG фиксируется в БД со следующими статусами:
* `STARTED`- проставляется в самом начале работы пайплайна (задача `init_etl_run`).
* `SUCCESS`— фиксируется автоматически через `on_success_callback`, если абсолютно все шаги графа завершились успешно.
* `FAILED`— фиксируется через `on_failure_callback` при падении любого таска. Дополнительно в поле `error_message` записывается детальный трейсбэк ошибки для администратора.

### Логика повторного запуска
* `depends_on_past = False`: Важнейшая настройка для батч-процессов. Если вчерашний запуск упал из-за сетевого сбоя, это не заблокирует запуск сегодняшнего батча. Пайплайн можно перезапускать независимо.
* `retries = 1` и `retry_delay = 1 минута`: В случае кратковременной потери связи с базой данных, Airflow не уронит весь пайплайн сразу, а подождет минуту и попробует выполнить упавший шаг еще раз.

## Этапы выполнения Airflow DAG

Пайплайн состоит из 6 последовательных шагов (задач):

1. **`init_etl_run`**: Инициализация. Создает запись в таблице логирования со статусом `STARTED`.
2. **`load_students_to_raw`** / **`load_student_marks_to_raw`**: Этап **Extract & Load**. Запускаются параллельно для экономии времени. Один модуль чистит и перегружает студентов, второй — дописывает новые оценки.
3. **`transform_raw_to_dwh`**: Этап **Transform**. Запускает SQL-скрипты/процедуры для переноса данных из сырого слоя в системное хранилище `dwh`.
4. **`data_quality_checks`**: Этап **Валидации**. Проверяет данные на критические ошибки (например: отсутствие `null` значений в ключевых полях, проверка, что оценки находятся в диапазоне от 1 до 5 и т.д.).
5. **`publish_to_marts`**: Этап **Publish**. Собирает проверенные данные в витрины слоя `mart` для конечных пользователей.
